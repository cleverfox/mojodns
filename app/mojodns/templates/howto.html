API tokens

Most automation around mojodns — listing your zones, issuing TLS certificates, dynamic DNS — is authenticated with an API token. A token acts on your behalf: it can only reach zones you own or edit. Pass it as the X-API-Key HTTP header (preferred) or as a ?token=… query parameter.

Create and revoke your tokens on the account page — pick a name and an expiry; the secret is shown once, at creation, and stored hashed thereafter. An expired token returns 401 token expired.

list zones available to your token

curl -H "X-API-Key: <token>" {{ base }}/api/v1/zones

TLS certificates (acme.sh / Let's Encrypt)

The panel is compatible with the stock dns_pdns module of acme.sh, so DNS-01 validation (including wildcard certificates) works without any custom plugins. Your token may only create and remove TXT records, in your zones only.

export PDNS_Url="{{ base }}"
export PDNS_ServerId="localhost"
export PDNS_Token="<token>"
export PDNS_Ttl=60

acme.sh --issue --dns dns_pdns -d example.com -d '*.example.com'

Validation records propagate to the public nameservers within a couple of seconds, so no extra --dnssleep is normally needed.

Dynamic DNS

Hosts with changing addresses can keep an A/AAAA record up to date. The endpoint is idempotent — if the address didn't change, nothing is rewritten — so calling it from cron every few minutes is fine.

raw API

# set/update a record (omit &ip=… to use the source address of the request)
curl -X POST -H "X-API-Key: <token>" \
  "{{ base }}/api/v1/ddns?name=home.example.com&type=A&ip=203.0.113.7&ttl=300"

# remove it
curl -X DELETE -H "X-API-Key: <token>" \
  "{{ base }}/api/v1/ddns?name=home.example.com&type=AAAA"

ready-made client (bash / FreeBSD sh)

Download mojodns-ddns.sh and mojodns-ddns.conf.example, then:

fetch {{ base }}/static/client/mojodns-ddns.sh   # or curl -O / wget
install -m 755 mojodns-ddns.sh /usr/local/bin/
curl -o /usr/local/etc/mojodns-ddns.conf {{ base }}/static/client/mojodns-ddns.conf.example
vi /usr/local/etc/mojodns-ddns.conf              # set PANEL_URL, TOKEN, RECORD

# every 5 minutes from cron:
*/5 * * * * /usr/local/bin/mojodns-ddns.sh

The client discovers the host's public IPv4/IPv6 with curl -4 / curl -6 and updates the records. Behaviour is controlled by:

UPDATE_A / UPDATE_AAAA which address families to manage (true/false)
DEL_STALE_A / DEL_STALE_AAAA what to do when that family has no connectivity: true — delete the now-stale record from DNS; false — leave it untouched (default)
IP4_URL / IP6_URL where to learn the public address (default https://ifconfig.me); set to auto to skip discovery and let the panel use the source address of the update request
TTL / CURL_TIMEOUT / QUIET record TTL (default 300), curl timeout, suppress normal output for cron

A safety detail: if the IPv6 path answers with an IPv4 address (NAT64 networks), the client treats it as "no IPv6" instead of publishing a broken AAAA record.

Dynamic updates (RFC 2136)

For tools that speak the standard DNS UPDATE protocol — nsupdate, certbot dns-rfc2136, Kubernetes external-dns, Terraform's dns provider — mojodns accepts RFC 2136 updates directly against the master at {% if master_host %}{{ master_host }}{% else %}<master>{% endif %}:53 (UDP + TCP). This is separate from the HTTP Dynamic-DNS endpoint above: no API token is involved — updates are authorised by a TSIG key (and, optionally, a source-IP allow-list) that you configure per zone.

enable it for a zone

Open the zone, go to settings → dynamic updates (RFC 2136), and generate a key (the panel shows the secret) or import one a tool gave you. Each zone keeps its own dedicated update keys, distinct from any transfer keys. Optionally add an IP/CIDR allow-list. A zone with no key and no allow-list rejects every update — nothing is open by default.

nsupdate

nsupdate -y 'hmac-sha256:example.com-update:<secret>' <<'EOF'
server {% if master_host %}{{ master_host }}{% else %}<master>{% endif %}

zone example.com.
update delete www.example.com. A
update add    www.example.com. 300 A 203.0.113.7
send
EOF

The algorithm, key name and secret must match exactly what the panel shows for the zone. Serials are bumped automatically and the change propagates to the public nameservers within a couple of seconds.

certbot (DNS-01 without acme.sh)

# /etc/letsencrypt/rfc2136.ini  (chmod 600)
dns_rfc2136_server  = {% if master_host %}{{ master_host }}{% else %}<master>{% endif %}

dns_rfc2136_port    = 53
dns_rfc2136_name    = example.com-update
dns_rfc2136_secret  = <secret>
dns_rfc2136_algorithm = HMAC-SHA256

certbot certonly --dns-rfc2136 \
  --dns-rfc2136-credentials /etc/letsencrypt/rfc2136.ini \
  -d example.com -d '*.example.com'

A TSIG key may only touch the single zone it's attached to, so issue a separate key per zone you want to manage this way.
